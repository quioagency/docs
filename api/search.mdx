The Search API lets you bring Company Brain's knowledge retrieval directly into your own applications. Send a plain-English question or keyword query, and receive ranked document snippets — optionally accompanied by an AI-generated answer with cited sources. Whether you are building a customer-facing chatbot, an internal search interface, or an automated Q&A pipeline, the Search API gives you the flexibility to choose exactly how results are surfaced.

## Search the knowledge base

### POST /api/v1/search

Submit a natural language or keyword query against your workspace's indexed content. Control the number of results, narrow the search to specific sources, and opt in to an AI-generated answer that synthesizes content across matching documents.

#### Body parameters

<ParamField body="query" type="string" required>
  The search query. Write this as a natural language question or a keyword phrase — for example, `"What is our parental leave policy?"` or `"parental leave entitlement weeks"`.
</ParamField>

<ParamField body="mode" type="string" default="semantic">
  The retrieval strategy to use. 

  - `semantic` — uses vector embeddings to find conceptually related content even when the exact words differ. Best for natural language questions.
  - `keyword` — performs exact and fuzzy token matching. Best when you know the precise terminology used in your documents.
</ParamField>

<ParamField body="limit" type="integer" default="5">
  The maximum number of document results to return. Minimum value is `1`; maximum is `20`.
</ParamField>

<ParamField body="source_ids" type="array">
  An optional array of source IDs to restrict the search to. When omitted, Company Brain searches across all connected sources and uploaded documents. Retrieve source IDs from the [Sources API](/api/sources).
</ParamField>

<ParamField body="include_answer" type="boolean" default="false">
  When `true`, Company Brain generates a concise AI answer synthesized from the top matching documents and includes it in the response alongside the ranked results. When `false`, only ranked results are returned.
</ParamField>

#### Example request

```bash
curl -X POST https://api.companybrain.ai/v1/search \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "What is our customer refund policy?",
    "mode": "semantic",
    "limit": 3,
    "include_answer": true
  }'
```

#### Example response

```json
{
  "data": {
    "answer": {
      "text": "Customers can request a refund within 30 days of purchase...",
      "confidence": "high",
      "sources": ["doc_01HXYZ"]
    },
    "results": [
      {
        "document_id": "doc_01HXYZ",
        "title": "Refund Policy",
        "snippet": "Customers can request a refund within 30 days of purchase for any reason.",
        "source_type": "notion",
        "url": "https://notion.so/...",
        "score": 0.97
      }
    ]
  }
}
```

#### Response fields

<ResponseField name="data" type="object">
  The top-level response envelope.

  <Expandable title="Response fields">
    <ResponseField name="answer" type="object">
      Present only when `include_answer` is `true` in the request.

      <Expandable title="Answer object fields">
        <ResponseField name="text" type="string">
          The AI-generated answer to your query, written in plain language and synthesized from the top matching documents.
        </ResponseField>

        <ResponseField name="confidence" type="string">
          Company Brain's confidence level in the generated answer. Possible values: `high`, `medium`, `low`. A `low` confidence score typically means the knowledge base does not contain a clear, authoritative answer to the question.
        </ResponseField>

        <ResponseField name="sources" type="array">
          An array of document IDs that were used to generate the answer. Cross-reference these with the `results` array to surface citations in your UI.
        </ResponseField>
      </Expandable>
    </ResponseField>

    <ResponseField name="results" type="array">
      An ordered array of matching document objects, ranked by relevance score descending.

      <Expandable title="Result object fields">
        <ResponseField name="document_id" type="string">
          The unique identifier of the matching document.
        </ResponseField>

        <ResponseField name="title" type="string">
          The title of the matching document.
        </ResponseField>

        <ResponseField name="snippet" type="string">
          A short excerpt from the document showing the most relevant passage for your query. Suitable for display in a search results list.
        </ResponseField>

        <ResponseField name="source_type" type="string">
          The integration that produced this document. Possible values: `notion`, `google_drive`, `slack`, `confluence`, `upload`.
        </ResponseField>

        <ResponseField name="url" type="string">
          The original URL of the document in its source system, if available. Link users directly to the source for full context.
        </ResponseField>

        <ResponseField name="score" type="number">
          A relevance score between `0.0` and `1.0`. Higher scores indicate stronger matches. Scores above `0.85` generally indicate a high-quality match.
        </ResponseField>
      </Expandable>
    </ResponseField>
  </Expandable>
</ResponseField>

## Choosing the right configuration

<Tip>
  Use `include_answer: true` when building chatbot or Q&A integrations where you want to surface a single, synthesized response — for example, a Slack bot that answers employee questions automatically. Use `include_answer: false` when building a search UI where you want to show users a list of ranked documents and let them choose which to read. This keeps latency lower and gives users direct access to source material.
</Tip>

<Tabs>
  <Tab title="Chatbot integration">
    When building a chatbot, set `include_answer: true` and display `data.answer.text` as the bot's response. Include links to `data.answer.sources` so users can verify the answer in the original document.

    ```json
    {
      "query": "How many vacation days do new employees get?",
      "mode": "semantic",
      "limit": 5,
      "include_answer": true
    }
    ```
  </Tab>
  <Tab title="Search UI">
    When building a search interface, set `include_answer: false` and render `data.results` as a list of cards showing `title`, `snippet`, `source_type`, and a link via `url`.

    ```json
    {
      "query": "vacation days new employee",
      "mode": "keyword",
      "limit": 10,
      "include_answer": false
    }
    ```
  </Tab>
  <Tab title="Scoped search">
    To search within a specific tool — for example, only Confluence pages — pass the relevant `source_ids`. Retrieve available source IDs from `GET /sources`.

    ```json
    {
      "query": "deployment runbook for production",
      "mode": "semantic",
      "limit": 5,
      "source_ids": ["src_01HCON1", "src_01HCON2"],
      "include_answer": true
    }
    ```
  </Tab>
</Tabs>

## Rate limits

Search requests count toward your plan's overall API rate limit. If you exceed your plan's limit, the API returns a `429 Too Many Requests` response with a `Retry-After` header indicating how many seconds to wait before retrying. To avoid hitting limits in high-traffic applications, consider caching answers for identical or semantically similar queries.

<Note>
  Requests with `include_answer: true` consume more compute and count as two units against your rate limit — one for retrieval and one for answer generation. If you need higher throughput, contact support to discuss a plan upgrade or dedicated capacity.
</Note>
