Webhooks let your systems react to Company Brain events the moment they happen, without polling the API. When a relevant event occurs in your workspace — a new document is indexed, a sync completes, or a question gets escalated to a human — Company Brain sends an HTTP POST request to the URL you register. Use webhooks to trigger downstream automation, invalidate caches, post notifications to Slack, or keep an external system synchronized with your knowledge base.

## Supported events

Company Brain can notify your endpoint about the following event types:

| Event | Description |
|---|---|
| `knowledge.created` | A new document was indexed from any source or uploaded via the API. |
| `knowledge.deleted` | A document was removed from the knowledge base. |
| `source.sync.completed` | A source finished syncing successfully. |
| `source.sync.failed` | A source sync encountered an error and did not complete. |
| `escalation.created` | A question was escalated to a human expert because the knowledge base could not answer it confidently. |

## Webhook payload format

Every webhook request Company Brain sends to your endpoint shares the same top-level structure. The `data` object varies depending on the event type.

```json
{
  "event": "knowledge.created",
  "timestamp": "2024-01-15T10:30:00Z",
  "workspace_id": "ws_01HABC",
  "data": {
    "document_id": "doc_01HXYZ",
    "title": "Updated Refund Policy",
    "source_id": "src_01HDEF"
  }
}
```

<ResponseField name="event" type="string">
  The name of the event that triggered this delivery. Use this field to route the payload to the correct handler in your application.
</ResponseField>

<ResponseField name="timestamp" type="string">
  ISO 8601 timestamp indicating when the event occurred in Company Brain.
</ResponseField>

<ResponseField name="workspace_id" type="string">
  The ID of the workspace in which the event occurred. Useful when a single endpoint receives webhooks from multiple Company Brain workspaces.
</ResponseField>

<ResponseField name="data" type="object">
  Event-specific payload. The fields present depend on the event type. For `knowledge.created` and `knowledge.deleted`, this includes `document_id`, `title`, and `source_id`. For `source.sync.completed` and `source.sync.failed`, it includes `source_id` and `source_name`. For `escalation.created`, it includes `escalation_id` and `query`.
</ResponseField>

## Verifying webhook signatures

Company Brain signs every webhook request with HMAC-SHA256 using the secret generated when you create the endpoint. The signature is sent in the `X-Brain-Signature` header as a hex-encoded digest of the raw request body. Always verify this signature before processing the payload to ensure the request genuinely originated from Company Brain.

<Steps>
  <Step title="Retrieve your webhook secret">
    Copy the `secret` value returned when you created the webhook endpoint. This secret is only shown once — if you did not store it, you must delete the webhook and create a new one.
  </Step>
  <Step title="Read the raw request body">
    Use the raw bytes of the incoming request body for the HMAC computation. Do not parse the JSON first — any whitespace or key-ordering differences introduced by a JSON parser will cause the signature check to fail.
  </Step>
  <Step title="Compute the expected signature">
    Generate an HMAC-SHA256 digest of the raw body using your secret, then compare it to the value in `X-Brain-Signature` using a constant-time comparison function to prevent timing attacks.
  </Step>
  <Step title="Reject mismatched signatures">
    Return a `401 Unauthorized` response if the signatures do not match. Do not process the payload.
  </Step>
</Steps>

<Tabs>
  <Tab title="Python">
    ```python
    import hmac
    import hashlib

    def verify_signature(payload_body: bytes, secret: str, signature: str) -> bool:
        expected = hmac.new(
            secret.encode(),
            payload_body,
            hashlib.sha256
        ).hexdigest()
        return hmac.compare_digest(expected, signature)
    ```
  </Tab>
  <Tab title="Node.js">
    ```javascript
    const crypto = require("crypto");

    function verifySignature(payloadBody, secret, signature) {
      const expected = crypto
        .createHmac("sha256", secret)
        .update(payloadBody)
        .digest("hex");
      return crypto.timingSafeEqual(
        Buffer.from(expected),
        Buffer.from(signature)
      );
    }
    ```
  </Tab>
  <Tab title="Ruby">
    ```ruby
    require "openssl"

    def verify_signature(payload_body, secret, signature)
      expected = OpenSSL::HMAC.hexdigest("sha256", secret, payload_body)
      ActiveSupport::SecurityUtils.secure_compare(expected, signature)
    end
    ```
  </Tab>
</Tabs>

## Create a webhook

<br />

### POST /webhooks

Register a new webhook endpoint and select which events should trigger deliveries to it. Company Brain immediately begins sending matching events to your URL.

#### Body parameters

<ParamField body="url" type="string" required>
  The publicly accessible HTTPS URL that Company Brain should POST events to. The endpoint must respond with a `2xx` status code within 10 seconds or Company Brain treats the delivery as failed.
</ParamField>

<ParamField body="events" type="array" required>
  An array of event names this webhook should receive. Pass `["*"]` to subscribe to all current and future events. Valid individual values: `knowledge.created`, `knowledge.deleted`, `source.sync.completed`, `source.sync.failed`, `escalation.created`.
</ParamField>

#### Example request

```bash
curl -X POST https://api.companybrain.ai/v1/webhooks \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://yourapp.example.com/webhooks/brain",
    "events": ["knowledge.created", "source.sync.completed", "source.sync.failed"]
  }'
```

#### Example response

```json
{
  "data": {
    "id": "wh_01HPQR",
    "url": "https://yourapp.example.com/webhooks/brain",
    "events": [
      "knowledge.created",
      "source.sync.completed",
      "source.sync.failed"
    ],
    "secret": "whsec_a3f8c2e1d94b7056f3e2a1c8d4b9e5f7a2c3d1e6"
  }
}
```

<Warning>
  The `secret` field is only included in the response to this creation request. Company Brain will never show it again. Copy it immediately and store it in a secure location such as your application's secrets manager or environment variables. If you lose the secret, you must delete the webhook and create a new one to obtain a fresh secret.
</Warning>

## List webhooks

<br />

### GET /webhooks

Retrieve all webhook endpoints registered for your workspace.

#### Example request

```bash
curl https://api.companybrain.ai/v1/webhooks \
  -H "Authorization: Bearer YOUR_API_KEY"
```

#### Example response

```json
{
  "data": [
    {
      "id": "wh_01HPQR",
      "url": "https://yourapp.example.com/webhooks/brain",
      "events": [
        "knowledge.created",
        "source.sync.completed",
        "source.sync.failed"
      ]
    }
  ]
}
```

<Note>
  The `secret` field is omitted from list responses for security. Only the creation response includes the secret.
</Note>

## Delete a webhook

<br />

### DELETE /webhooks/{id}

Remove a webhook endpoint. Company Brain immediately stops sending events to the associated URL.

#### Path parameters

<ParamField path="id" type="string" required>
  The unique ID of the webhook to delete. Retrieve this from `GET /webhooks`.
</ParamField>

#### Example request

```bash
curl -X DELETE https://api.companybrain.ai/v1/webhooks/wh_01HPQR \
  -H "Authorization: Bearer YOUR_API_KEY"
```

#### Response

Returns `204 No Content` on success with an empty response body.

## Retry behavior

If your endpoint does not respond with a `2xx` status code within 10 seconds, Company Brain marks the delivery as failed and retries it automatically. Retries follow an exponential backoff schedule:

| Attempt | Delay after previous attempt |
|---|---|
| 1st retry | 1 minute |
| 2nd retry | 5 minutes |
| 3rd retry | 30 minutes |

After three failed retries, Company Brain stops attempting to deliver that event. Failed deliveries are visible in the **Webhooks** section of your dashboard, where you can inspect the response your endpoint returned and manually replay any delivery.

<Tip>
  To avoid missed events during planned downtime or deployments, consider temporarily disabling your webhook endpoint from the dashboard rather than letting deliveries fail. Paused webhooks queue events and resume delivery once you re-enable the endpoint — up to a maximum queue age of 72 hours.
</Tip>
