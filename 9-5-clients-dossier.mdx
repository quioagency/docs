### 1. 35 Felder sind bereits zur Löschung markiert (`_DELETE_`) — endgültig weg in Phase 11. Bitte bestätigen.

Diese kannst du in einem Rutsch durchwinken — sie sind aus Phase 08/09 markiert, haben (bis auf Datenreste) keinen aktiven Code mehr:

- **Follower-Zahlen (8):** ig/tiktok/yt/fb/linkedin/threads/x\_followers, followers\_updated\_at
- **Alte Social-URLs (3):** instagram\_url, youtube\_url, linkedin\_url _(ersetzt durch ig\_username, yt\_channel\_url, linkedin\_company\_url)_

- **❓ Eine Rückfrage hier:** `special_terms` war mal als „aktiv genutzt" (FROXIMUN-Referral) eingestuft — jetzt markiert. Wirklich weg, oder doch behalten?

### 2. Paket wird in **4 Feldern gleichzeitig** geführt — das ist zu viel.

`package` (92%), `package_slug` (3%), `deal_package` (2%), `package_link` (3%). Du pflegst `package_slug` seit Phase 09 manuell, der Code liest aber teils noch `package`/`deal_package`. **Frage:** Welches ist künftig das Eine? (Vorschlag: `package_slug` als Wahrheit, Rest aufräumen.)

### 3. Brand-Assets werden hochgeladen, aber von keiner Automation gelesen.

`brand_logo`, `brand_guide`, `profile_picture` (alle 0%), `brand_colors`, `brand_fonts` (0%), `download_links` (0%) — werden von der nativen Upload-Automatik **geschrieben**, aber **kein Code liest sie**. **Frage:** Sollen die ins Interface (Team schaut Logo/Guide an) — oder sind sie totes Gewicht? Das ist genau eine Interface-Entscheidung.

### 4. Doppelung Start-Datum: `start_date` (91%) vs. `contract_start` (91%).

Zwei fast identisch gefüllte Datumsfelder. `contract_start` wird vom Code genutzt, `start_date` nur an einer Stelle. **Frage:** Zusammenlegen?

### 5. ~22 leere Felder (0% befüllt), nicht markiert — wozu existieren sie?

Zwei Untergruppen:

- **Erwartbar leer (Daten kommen erst bei neuen Deals):** close\_deal\_id, docusign\_envelope\_id, contract\_signed, welcome\_mail\_sent\_at, clientid, cks\_dirty → **behalten** (aktiv in Sales/Onboarding-Code).
- **Echte Kandidaten zum Wegräumen:** backoffice\_email, invoice\_email, slack\_email, additional\_contacts, additional\_emails, billing\_state, threads\_username, x\_username, onboarding\_guide\_task\_id → 0% \+ kaum/kein Code. **Behalten oder weg?**

### 6. 10 Zeit-Schätzungs-Felder (`te_*`) — Backend oder Interface?

te\_strategy\__, te\_production\__, te\_qc, te\_scheduling, te\_revision (je 38%) — speisen die Subtask-Zeitschätzungen. Aktiv, aber: gehören die ins Team-Interface oder bleiben sie reines Backend?

## Methodik (Tech-Fußnote)

- **Live-Daten** (Typ, Optionen, Füllgrad, `_DELETE_`-Status): Meta-API \+ alle 65 Records, 2026-06-22. Roh: `_raw-clients-live-inventory.md`.
- **Code-/Automations-Bezug:** frischer Grep über das gesamte `agents/` (Feld-Name **und** Field-ID-Konstante, ADR-478), Quote-Filter gegen Namens-Kollisionen (z.B. `status` als lokale Variable), Tests/Skripte separat gezählt; native \[\[Airtable\]\]-Automationen aus `native-automation-inventur.md`. Roh: `_raw-clients-code-map.md`. **Nicht** aus der stale cross-workstream-field-map.
- „Genutzt von" ist verdichtet — die vollständige Modul-Liste pro Feld steht in der Roh-Datei.
- Befüllung über 65 Kunden; bei Feldern, die sich erst bei neuen Deals/Onboardings füllen, ist 0% normal (im Status vermerkt).
